## 一、SAG 是什么

SAG 是一个**基于图结构的下一代 RAG 系统**，专门为 LLM Agent 设计。其核心创新是用 **chunk → event → entity** 三元组替代传统的"分块→向量"范式：

- **Event（事件）**：一个完整的语义单元（比如一段完整的论述）
- **Entity（实体）**：从事件中提取的关键概念/名词
- **双向链接**：event ↔ entity 互相索引，支持多跳查询

检索时，从匹配的事件出发，通过实体关系链进行多跳扩展，不需要维护重型知识图谱。

### 技术栈

| 层 | 技术 |
|---|------|
| 前端 | React + Vite + Tailwind CSS + @xyflow/react（知识图谱可视化） |
| 后端 | Fastify（HTTP API）+ MCP SDK（Agent 工具接口） |
| 数据库 | PostgreSQL + pgvector（向量 + 全文搜索 + SQL 多跳查询） |
| 模型 | OpenAI 兼容接口（LLM + Embedding + Rerank） |
| 语言 | 纯 TypeScript，Node.js 20+ |

### 支持的文档格式

仅支持 **Markdown (.md)** 和 **纯文本 (.txt)**。不支持 JSON 或其他结构化格式。

## 二、当前五运六气 RAG 架构对比

| 维度 | 当前架构（7 asset JSON） | SAG |
|------|------------------------|-----|
| **数据格式** | 结构化 JSON（键值对数组） | 自由文本 Markdown/TXT |
| **检索方式** | 精确 Key 匹配（确定性） | 向量相似度 + 图多跳（概率性） |
| **基础设施** | 零 — 纯文件系统 | PostgreSQL + pgvector（需 Docker/安装） |
| **语言栈** | Python | TypeScript |
| **知识量** | 104 个唯一键，7 个 JSON | 适合大规模文档集（百/千级） |
| **集成方式** | Agent 直接读文件 | MCP 协议 / HTTP API |
| **确定性** | 100% — 给定 key 返回固定结果 | 概率性 — 语义相似度检索 |
| **部署复杂度** | 极低 | 较高（数据库 + Node.js 服务） |

## 三、核心评估：能不能引入？

### 3.1 作为替代方案 → 不推荐

**当前 7 个 asset JSON 不适合用 SAG 替代**，原因：

1. **结构化 vs 非结构化**：JSON 中的每个条目是经过精心组织的结构化知识（如 `water_excess` 的病机描述包含 17 个字段），SAG 的 Markdown 分块方式会丢失结构信息
2. **确定性需求**：临床病机分析要求精确匹配——"水运太过的病机是什么？"必须返回正确的条目，而不是"语义上相似"的条目。SAG 的概率性检索在医疗场景中是风险
3. **JSON 非原生支持**：SAG 只处理 Markdown/TXT，JSON 文件需要额外适配器
4. **过度设计**：104 个键的规模用不上图数据库和向量检索

### 3.2 作为补充方案 → 有条件推荐

**SAG 适合处理「自由文本文献层」**，即当前架构的薄弱环节：

| 适合 SAG 的场景 | 不适合 SAG 的场景 |
|----------------|-----------------|
| 经典原文全文（素问七篇大论完整章节） | 岁运病机键值对（asset1） |
| 历代注家长篇论述（如张景岳《类经》全文） | 三因司天方结构化数据（asset4） |
| 临床医案记录（自由叙事文本） | 地域修正参数（asset6） |
| 现代研究论文摘要 | 体质调理方案结构化条目（asset7） |
| 灵枢九宫八风等参考文献 | 客主加临 key 映射 |

### 3.3 混合架构建议

```
┌─────────────────────────────────────────────────┐
│                 LLM Agent                         │
├─────────────────────────────────────────────────┤
│  ① 精确 RAG (现有 7 Asset JSON)                  │
│     └─ 岁运/方剂/注家/地域/体质 → 确定性查找      │
│                                                     │
│  ② 语义 RAG (SAG — 可选扩展)                      │
│     └─ 经典全文/医案/论文 → 图检索 + 向量相似度   │
├─────────────────────────────────────────────────┤
│  MCP 协议桥接 (SAG 自带 MCP Server)               │
│  HTTP API 桥接 (Python → TypeScript)              │
└─────────────────────────────────────────────────┘
```

## 四、引入 SAG 的代价

### 必要投入

| 项目 | 说明 |
|------|------|
| PostgreSQL 安装 | Windows 需安装 PostgreSQL 16+，启用 pgvector 扩展 |
| Node.js 20+ | 需安装或确认已有 |
| SAG 服务部署 | clone → npm install → 配置 .env → 数据库迁移 → 启动 |
| JSON 适配器开发 | 写一个 Python/TypeScript 脚本，将结构化 JSON 转为 SAG 可 ingest 的 Markdown |
| 双 RAG 路由 | Agent 需要判断何时用精确 RAG、何时用 SAG 语义检索 |
| API 桥接 | Python 端通过 HTTP 调用 SAG 的 `/api/search` 端点 |

### 持续运维

- PostgreSQL 数据库备份与维护
- SAG 服务进程管理（开机自启、异常重启）
- Embedding 模型 API 费用（每次 ingest 和 search 都调用）
- 当文献量增长时，可能需要更新向量索引

## 五、分阶段建议

### 第一阶段（现在 — 当前架构已够用）

维持现有 7 asset JSON 架构，因为它已经：
- 覆盖了 5 层知识（104 唯一键）
- 67/67 验证全部通过
- 零基础设施依赖
- 完全确定性检索

短期内如果要扩展文献，使用 `ingest_literature.py` 继续添加结构化 asset JSON 即可。

### 第二阶段（当文献量 > 200 篇时考虑引入 SAG）

当需要注入以下类型的大量文献时，SAG 的价值才体现出来：
- 全文经典（素问 81 篇、灵枢 81 篇全部录入）
- 历代注家著作全文（如《类经》《素问吴注》《素问绍识》等）
- 数百份临床医案
- 现代运气学研究论文库（PubMed 文献）

### 第三阶段（理想混合架构）

```
Agent 推理流程:

1. 用户提问
2. 提取 RAG key (如 water_excess)
3. ① 精确 RAG: 查询 asset JSON → 命中 → 直接返回
4. 若精确 RAG 未覆盖:
   └─ ② 语义 RAG: 调用 SAG MCP 工具搜索
      ├─ sag_search("水运太过 病机 王冰")
      ├─ 返回 events + entities + trace
      └─ Agent 综合推理
5. 自进化记录 (self_evolve.py)
```

## 六、结论

| 问题 | 回答 |
|------|------|
| **可以完全替换当前 RAG 吗？** | ❌ 不推荐 — 架构不匹配，会降低确定性 |
| **可以作为补充引入吗？** | ✅ 有条件可以 — 专用于自由文本文献层 |
| **现在应该引入吗？** | ⏸️ 建议暂缓 — 当前 104 键规模不值得引入全套基础设施 |
| **什么时机引入合适？** | 当文献量 > 200 篇或需要全文经典检索时 |
| **最快集成路径？** | Docker 启动 PostgreSQL → 部署 SAG → 写 JSON→Markdown 转换器 → Agent 通过 MCP 调用 |

### 简版结论

SAG 是一个优秀的大规模自由文本图检索系统，但当前 wuyun-liuqi 项目的 7 asset JSON 精确 RAG 架构与它的设计目标不同。**不建议现在引入**，但值得关注和记录这个方案，在知识规模扩大 10 倍以上时重新考虑。
